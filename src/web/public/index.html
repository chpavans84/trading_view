✅ Trade Entry Checklist

Steps the bot runs on every scan (in order):
  1. Daily P&L → stop if target hit or loss limit reached
  2. Market status → skip if closed

🔮 5-Algorithm Stock Predictor
Pure JavaScript, zero LLM cost. All 5 algorithms run in parallel via Promise.allSettled, results cached 15 minutes per symbol. Triggered when Claude calls the get_stock_prediction tool.
ALGO 1
Linear Regression Trend
Uses last 50 daily bars from Yahoo Finance. Computes least-squares slope, intercept, R² (coefficient of determination), and standard error of residuals.
Projects the regression line forward: day-5 and day-10 price targets.
Confidence band = ±1.5 × std error of residuals (as % of price).
R² > 0.7 = high reliability  ·  R² 0.4–0.7 = medium  ·  R² < 0.4 = choppy/unreliable
ALGO 2
ATR Expected Move
14-period Wilder ATR from same OHLCV bars (True Range = max of H-L, |H-PrevClose|, |L-PrevClose|).
Probability ranges: ±1.0×ATR (1 day) · ±2.2×ATR (5 days, ≈√5) · ±3.2×ATR (10 days, ≈√10).
ATR% = ATR ÷ price. ATR% > 3% = high volatility warning.
ALGO 3
Momentum Score (0–100)
Five sub-signals summed (−10 to +10 raw), then normalized to 0–100:
  • RSI(14): 50–70 = +2 (bullish) · >70 = −1 (overbought) · 30–50 = −2 (bearish) · <30 = +1 (oversold)
  • EMA9/20/50 alignment: price>EMA9>EMA20>EMA50 = +3 (fully bullish) · reverse = −3 · partial = ±1 per aligned EMA
  • MACD(12,26,9): line > signal AND histogram growing = +2 · below and shrinking = −2
  • Volume trend: 5-day avg > 20-day avg (accumulation) = +1 · else −1 (distribution)
  • Price vs 20-day SMA (VWAP proxy): above = +1 · below = −1
Labels: Strong Bullish ≥70 · Bullish ≥55 · Neutral ≥45 · Bearish ≥30 · Strong Bearish <30
ALGO 4
Personal Trade Edge
Queries the trades table for all closed trades on this symbol.
Computes: win rate · avg win/loss P&L · profit factor (sum wins ÷ sum losses) · avg hold time · best hour of day · best day of week · last 3 trades.
Edge label: Strong Edge (win rate > 60% AND profit factor > 1.5) · Slight Edge (win rate > 50%) · No Edge / Need More Data.
Requires ≥ 3 closed trades on this symbol. Returns insufficient_data: true otherwise.
ALGO 5
Earnings Catalyst Model
Reads last 8 quarters from the fundamentals table (revenue, net income, EPS).
Revenue growth trend: QoQ deltas → accelerating / decelerating / flat.
Earnings momentum score (0–100): revenue accelerating (+2) · net income positive & growing (+2) · EPS grew in 3+ of 4 quarters (+3) · normalized 0–100.
Next earnings date: yahoo-finance2 → calendarEvents.earnings.earningsDate (handles crumb auth).
Pre-earnings setup flags: "Pre-earnings long candidate" (≤14 days + momentum ≥60) · "Earnings imminent — high risk" (≤3 days).
Combined Signal Formula
overall_signal = 50
  + (momentum_score − 50) × 0.30   // momentum weight 30%
  + (trend_direction == "up" ? +15 : −15) // trend direction 15%
  + (R² × 10)                      // trend reliability bonus
  + (earnings_momentum − 50) × 0.20 // earnings weight 20%
  + (personal_win_rate > 0.5 ? +10 : −5) // personal edge
  clamped 0–100

🎯 Calibration Layer — Applied After Prediction
After the 5 algorithms produce a raw prediction, applyCalibration(symbol, changePct, rSquared) adjusts it using learned error patterns:
  • Bias correction — additive offset (e.g. MU consistently under-predicts by +3%). Weight scales 0→1 from 0→10 samples.
  • R² reversal damper — if R²>0.6 AND direction accuracy<35%, multiply prediction by 0.5. Rationale: high-R² trends are being over-extrapolated; the stock then reverses sharply.
  • Confidence score 0–100 — error score (50%) + direction accuracy (40%) + sample bonus (10%).
The same bias + reversal factors are re-applied to each forecast day via applyCalibrationToDay() with no additional DB calls.
Claude sees the calibration{} block on every get_stock_prediction tool call.
🎯 Prediction Calibration
A reinforcement learning layer that learns from every filled actual price and adjusts future predictions. Zero LLM cost — pure statistical correction on top of the 5-algorithm predictor.
⚠️ Key Discovery from Your Data
High R² stocks are the worst performers, not the best.
R²>0.6 (strong trend) stocks have only ~14% direction accuracy and the highest average error (~3.85%).
Cause: strong linear trends get over-extrapolated by the regression. The stock then mean-reverts or reverses sharply.
Fix: the calibration layer damps high-R² predictions by 50% when direction accuracy is below 35%.
STEP 1 — trainCalibration() — When it runs
Triggered automatically after every fillTodayActuals() call (EOD cron 4:15 PM ET + manual Fill Actuals button).
Requires ≥5 actuals to train — skips gracefully with a log message if insufficient data.
Scans all rows in stock_predictions where actual_price IS NOT NULL.
STEP 2 — Per-symbol factors stored in prediction_calibration
bias — mean(actual_change − predicted_change). Positive = model under-predicts. Applied additively, weighted by sample_size/10.
vol_scale — mean(|actual_change| / max(|predicted_change|, 0.1)). Captures whether real moves are larger than modeled.
dir_accuracy — fraction of predictions where direction (UP/DOWN) was correct.
STEP 3 — Global R²-bucket stats stored in prediction_calibration_global
Four buckets: r2_low (<0.1) · r2_poor (0.1–0.3) · r2_mid (0.3–0.6) · r2_high (>0.6).
For each: avg_abs_error and dir_accuracy stored.
Also stores model_bullish_bias (% of predictions that were UP) and actual_bullish_rate (% of stocks that actually closed UP).
STEP 4 — Individual errors logged to prediction_errors
One row per (symbol, target_date) with: r_squared, algorithm_signal, predicted_change_pct, actual_change_pct, error_pct, direction_correct.
Unique constraint on (symbol, target_date) — re-running trainCalibration() is idempotent.
Powers the Failure Analysis panel in Forecast Lab.
applyCalibration() — Applied at Forecast Generation
1. Load per-symbol bias (if ≥2 samples)
   weight = min(1, sample_size / 10)
   adjusted += bias × weight              // additive, constant across all days

2. Determine R² bucket → load dir_accuracy + expected_error

3. R² reversal damper (if R²>0.6 AND dir_acc<0.35):
   adjusted = adjusted × 0.5             // multiplicative, constant across all days

4. confidence = errorScore×0.5 + dirScore×0.4 + sampleBonus×0.1
   (0–100; green≥60, orange≥40, red<40)

For days 2–5 of the weekly forecast:
   applyCalibrationToDay(projPct, factors)
   → same bias + same reversal, no extra DB calls
Where calibration data surfaces

●Forecast Lab — Failure Analysis panel — worst symbols by error, R² bucket breakdown, direction quadrant counts, global bullish bias
●AI Chat get_stock_prediction — Claude receives calibration: { adjusted_change_pct, confidence, expected_error_pct, notes }
●GET /api/forecast — every day object includes adjusted_change_pct and confidence
●GET /api/forecast/failure-analysis — full failure breakdown JSON

🎯 How the 0–100 Conviction Score is Built
Every call to getConvictionScore(symbol) fires 14 data fetches in parallel, scores 20+ individual factors, applies ML-trained grade adjustments, and returns a single 0–100 number in ~2–4 seconds. Here is every detail.
Phase 1 · Parallel Fetch
14 data calls fire simultaneously
All wrapped in Promise.allSettled — any single failure scores zero for that factor, never aborts the whole score.
Phase 2 · Factor Scoring
20+ additive point buckets
Each signal maps to a fixed point value. TV chart factors are only applied when TradingView is live and are scaled by ML feature weights.
Phase 3 · ML Adjustment
Backtest alpha applied per grade
Grade computed from raw score → getFactorWeights() adds backtest alpha adj (e.g. A→+8, B→+3, C→-2, F→-9) → final score recalculated.
Phase 1 — 14 Parallel Data Fetches (Promise.allSettled)

After Phase 2A + 2B are summed, an interim grade is computed. Then getFactorWeights() loads the trained model weights from the model_results PostgreSQL table and applies a per-grade alpha adjustment.
Interim grade assignment
raw ≥ 80 → Grade A
raw ≥ 60 → Grade B
raw ≥ 40 → Grade C
raw < 40 → Grade F
Backtest alpha adj (typical values)
Grade A → +8 pts (historically outperforms SPY)
Grade B → +3 pts
Grade C → −2 pts
Grade F → −9 pts (historically underperforms)
Source: getFactorWeights() loads from model_results table (written by npm run research:train). The trained logistic regression model also provides individual feature weights (fw.rsi_norm, fw.ema_above, fw.macd_sign) that scale the TV-only factors in Phase 2B.

Fallback: If no model has been trained yet, weightsSource = 'none' and the adjustment is 0. The backtest_alpha_adj key is omitted from the breakdown entirely.

Re-computation: After adding backtest_alpha_adj to the breakdown, all values are summed again and clamped 0–100. The final grade is recomputed from the final score — it may differ from the interim grade.

Claude Code spawns the MCP server as a child process and communicates over stdin/stdout. No network port. Messages are newline-delimited JSON-RPC 2.0 frames.
Chrome DevTools Protocol (CDP)
MCP server connects to TradingView's Electron renderer via WebSocket. Executes JavaScript directly in the charting page — reads state, triggers actions, injects Pine Script.
Tool Execution Model
Claude calls a tool → MCP server receives JSON-RPC request → runs CDP Runtime.evaluate in Electron → returns structured result back to Claude.

📋 How to get your API credentials
  1. Open the Tiger Trade app (mobile or desktop) and log in to your account.
  2. Go to Profile → Settings → Developer or visit developer.itigerup.com and sign in with your Tiger account.
  3. Click Create Application (or open your existing app).
  4. Download the tiger_openapi_config.properties file.
  5. Open the file — you will find:
    • tiger_id → your Tiger ID
    • account → your Account Number
    • private_key_pk1 → your Private Key
⚠️ Use private_key_pk1 (not pk8). Your keys are stored securely and never sent to any third party.
